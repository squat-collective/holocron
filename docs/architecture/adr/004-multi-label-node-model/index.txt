14:T5b30,<h1 id="adr-004-multi-label-node-model-in-neo4j">ADR-004: Multi-Label Node Model in Neo4j<a class="anchor" aria-label="Link to section" href="#adr-004-multi-label-node-model-in-neo4j">#</a></h1>
<p><strong>Date:</strong> 2026-01-27
<strong>Status:</strong> Accepted
<strong>Deciders:</strong> Tom</p>
<h2 id="context">Context<a class="anchor" aria-label="Link to section" href="#context">#</a></h2>
<p>Holocron tracks different types of entities:</p>
<ul>
<li><strong>Assets:</strong> Dataset, Report, Process, System</li>
<li><strong>Actors:</strong> Person, Group</li>
</ul>
<p>We need to decide how to model these types in Neo4j.</p>
<h2 id="decision">Decision<a class="anchor" aria-label="Link to section" href="#decision">#</a></h2>
<p><strong>Use multiple labels per node: a base label + type-specific label.</strong></p>

<h2 id="options-considered">Options Considered<a class="anchor" aria-label="Link to section" href="#options-considered">#</a></h2>
<h3 id="1-single-label--type-property">1. Single Label + Type Property<a class="anchor" aria-label="Link to section" href="#1-single-label--type-property">#</a></h3>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="cypher" data-theme="github-dark-dimmed github-light"><code data-language="cypher" data-theme="github-dark-dimmed github-light" style="display: grid;"><span data-line=""><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">(:Asset </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">{</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">type</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">:</span><span style="--shiki-dark:#96D0FF;--shiki-light:#032F62"> "dataset"</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">,</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> ...</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">}</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">)</span></span>
<span data-line=""><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">(:Asset </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">{</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">type</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">:</span><span style="--shiki-dark:#96D0FF;--shiki-light:#032F62"> "report"</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">,</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> ...</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">}</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">)</span></span></code></pre></figure>
<p><strong>Pros:</strong></p>
<ul>
<li>Simple queries: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>MATCH (a:Asset)</span></span></code></span></li>
<li>Easy to add new types (no schema change)</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>Can't use label-specific indexes</li>
<li>Queries for specific type: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>WHERE a.type = 'dataset'</span></span></code></span> (slower)</li>
<li>No Neo4j-level type enforcement</li>
</ul>
<h3 id="2-separate-labels-only">2. Separate Labels Only<a class="anchor" aria-label="Link to section" href="#2-separate-labels-only">#</a></h3>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="cypher" data-theme="github-dark-dimmed github-light"><code data-language="cypher" data-theme="github-dark-dimmed github-light" style="display: grid;"><span data-line=""><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">(:Dataset </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">{</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">...</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">}</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">)</span></span>
<span data-line=""><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">(:Report </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">{</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">...</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">}</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">)</span></span>
<span data-line=""><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">(:Person </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">{</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">...</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">}</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">)</span></span></code></pre></figure>
<p><strong>Pros:</strong></p>
<ul>
<li>Clean separation</li>
<li>Type-specific indexes</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>Hard to query "all assets" without UNION</li>
<li>No shared base for common operations</li>
<li>Relationships must list all types</li>
</ul>
<h3 id="3-multiple-labels--selected">3. Multiple Labels ✅ Selected<a class="anchor" aria-label="Link to section" href="#3-multiple-labels--selected">#</a></h3>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="cypher" data-theme="github-dark-dimmed github-light"><code data-language="cypher" data-theme="github-dark-dimmed github-light" style="display: grid;"><span data-line=""><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">(:Asset:Dataset </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">{</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">...</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">}</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">)</span></span>
<span data-line=""><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">(:Asset:Report </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">{</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">...</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">}</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">)</span></span></code></pre></figure>
<p><strong>Pros:</strong></p>
<ul>
<li>Query all assets: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>MATCH (a:Asset)</span></span></code></span></li>
<li>Query specific type: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>MATCH (d:Dataset)</span></span></code></span></li>
<li>Label-specific indexes work</li>
<li>Neo4j-native pattern</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>Must remember to add both labels</li>
<li>Slightly more complex node creation</li>
</ul>
<h2 id="rationale">Rationale<a class="anchor" aria-label="Link to section" href="#rationale">#</a></h2>
<p>This is <strong>Neo4j's recommended pattern</strong> for type hierarchies. It gives us:</p>
<h3 id="1-flexible-querying">1. Flexible Querying<a class="anchor" aria-label="Link to section" href="#1-flexible-querying">#</a></h3>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="cypher" data-theme="github-dark-dimmed github-light"><code data-language="cypher" data-theme="github-dark-dimmed github-light" style="display: grid;"><span data-line=""><span style="--shiki-dark:#768390;--shiki-light:#6A737D">// All assets</span></span>
<span data-line=""><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">MATCH</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> (a:Asset) </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">RETURN</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> a</span></span>
<span data-line=""> </span>
<span data-line=""><span style="--shiki-dark:#768390;--shiki-light:#6A737D">// Only datasets</span></span>
<span data-line=""><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">MATCH</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> (d:Dataset) </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">RETURN</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> d</span></span>
<span data-line=""> </span>
<span data-line=""><span style="--shiki-dark:#768390;--shiki-light:#6A737D">// Assets owned by someone</span></span>
<span data-line=""><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">MATCH</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> (p:Person)</span><span style="--shiki-dark:#6CB6FF;--shiki-light:#005CC5">-</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">[:</span><span style="--shiki-dark:#F69D50;--shiki-light:#6F42C1">OWNS</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">]</span><span style="--shiki-dark:#6CB6FF;--shiki-light:#005CC5">-></span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">(a:Asset) </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">RETURN</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> a</span></span></code></pre></figure>
<h3 id="2-efficient-indexing">2. Efficient Indexing<a class="anchor" aria-label="Link to section" href="#2-efficient-indexing">#</a></h3>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="cypher" data-theme="github-dark-dimmed github-light"><code data-language="cypher" data-theme="github-dark-dimmed github-light" style="display: grid;"><span data-line=""><span style="--shiki-dark:#768390;--shiki-light:#6A737D">// Index on all assets</span></span>
<span data-line=""><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">CREATE</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> INDEX FOR (a:Asset) ON (a.name)</span></span>
<span data-line=""> </span>
<span data-line=""><span style="--shiki-dark:#768390;--shiki-light:#6A737D">// Index on datasets only</span></span>

<h2 id="consequences">Consequences<a class="anchor" aria-label="Link to section" href="#consequences">#</a></h2>
<h3 id="positive">Positive<a class="anchor" aria-label="Link to section" href="#positive">#</a></h3>
<ul>
<li>Idiomatic Neo4j usage</li>
<li>Efficient queries at both levels</li>
<li>Easy to extend with new types</li>
<li>Clear data model</li>
</ul>
<h3 id="negative">Negative<a class="anchor" aria-label="Link to section" href="#negative">#</a></h3>
<ul>
<li>Must ensure both labels are set on creation</li>
<li>Queries must use correct label for intent</li>
</ul>
<h3 id="mitigations">Mitigations<a class="anchor" aria-label="Link to section" href="#mitigations">#</a></h3>
<ul>
<li>Repository layer enforces label assignment</li>
<li>Type enums in Python ensure valid types</li>
<li>Integration tests verify label correctness</li>
</ul>
