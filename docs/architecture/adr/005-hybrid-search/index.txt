14:T2c54,<h1 id="adr-005-hybrid-vector--fulltext-search">ADR-005: Hybrid vector + fulltext search<a class="anchor" aria-label="Link to section" href="#adr-005-hybrid-vector--fulltext-search">#</a></h1>
<p><strong>Date:</strong> 2026-04-25
<strong>Status:</strong> Accepted
<strong>Deciders:</strong> Tom</p>
<h2 id="context">Context<a class="anchor" aria-label="Link to section" href="#context">#</a></h2>
<p>Holocron needs catalog-wide search that finds the right entity whether the user types:</p>
<ul>
<li>a literal substring of a column name (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>customer_id</span></span></code></span>),</li>
<li>a fuzzy concept (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>who owns the orders mart</span></span></code></span>),</li>
<li>a quoted phrase (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>"Q4 sales"</span></span></code></span>),</li>
<li>a graph constraint (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>owned by Tom and feeding the revenue mart</span></span></code></span>).</li>
</ul>
<p>Pure keyword search misses semantic intent. Pure vector search misses literal matches and is awful at "find me the column literally called <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>email</span></span></code></span>". Both give bad results when the catalog has many similarly-named entities (asset → schema container → field).</p>
<p>We need ranking that combines both signals, doesn't drown parent assets in their own columns, and has a small DSL for power users.</p>
<h2 id="decision">Decision<a class="anchor" aria-label="Link to section" href="#decision">#</a></h2>
<p>Run both rankers per kind and <strong>fuse their scores with a probabilistic OR</strong>:</p>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="plaintext" data-theme="github-dark-dimmed github-light"><code data-language="plaintext" data-theme="github-dark-dimmed github-light" style="display: grid;"><span data-line=""><span>combined = 1 − (1 − v_score) × (1 − fts_score)</span></span></code></pre></figure>
<p>On top of that:</p>
<ul>
<li><strong>Per-kind discounts</strong> so schema nodes don't dominate their parents:
<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>container × 0.85</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>field × 0.80</span></span></code></span>.</li>
<li><strong>Vector-only penalty</strong> when fulltext is dominant: if any FTS hit ≥ <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>0.55</span></span></code></span>, all pure-vector hits are scaled by <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>0.55</span></span></code></span>. This prevents a soft-matching asset from outranking a literal-matching field.</li>
<li><strong>Per-kind cap</strong> of 20 hits to keep response sizes bounded.</li>
<li><strong>Tiebreak by kind order</strong> so deterministic results for equal scores.</li>
</ul>
<p>Embeddings: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>BAAI/bge-small-en-v1.5</span></span></code></span> (384-dim, cosine) via fastembed (ONNX, CPU-only, ~300 MB resident, ~1 s warm-up). Indexed in Neo4j vector indexes alongside Lucene fulltext indexes on <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>name + description</span></span></code></span>.</p>
<p>DSL: bare words = semantic + fulltext; prefixes (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>ds:</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>f:</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>r:</span></span></code></span>) scope kind; named operators (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>owner:</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>feeds:</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>rule_for:</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>member:</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>uses:</span></span></code></span>) resolve to UID sets via fulltext and constrain results; <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>"phrase"</span></span></code></span> requires literal substring; <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>-term</span></span></code></span> excludes.</p>
<h2 id="options-considered">Options considered<a class="anchor" aria-label="Link to section" href="#options-considered">#</a></h2>
<h3 id="1-pure-keyword-lucene-only">1. Pure keyword (Lucene only)<a class="anchor" aria-label="Link to section" href="#1-pure-keyword-lucene-only">#</a></h3>
<ul>
<li><strong>Pros:</strong> simple, fast, no model download, works offline.</li>
<li><strong>Cons:</strong> misses semantic intent, awful at typos and synonyms, no relevance to query meaning.</li>
</ul>
<h3 id="2-pure-vector">2. Pure vector<a class="anchor" aria-label="Link to section" href="#2-pure-vector">#</a></h3>
<ul>
<li><strong>Pros:</strong> semantic match, ranks "concept matches" well.</li>
<li><strong>Cons:</strong> terrible at literal lookups (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>SELECT email FROM ...</span></span></code></span>), terrible at proper nouns, returns lots of "kinda close" garbage when nothing exists.</li>
</ul>
<h3 id="3-external-search-engine-elasticsearch--opensearch--qdrant">3. External search engine (Elasticsearch / OpenSearch / Qdrant)<a class="anchor" aria-label="Link to section" href="#3-external-search-engine-elasticsearch--opensearch--qdrant">#</a></h3>
<ul>
<li><strong>Pros:</strong> purpose-built, more knobs.</li>
<li><strong>Cons:</strong> extra service to operate, sync to maintain, second source of truth. Not justified for the catalog sizes Holocron targets.</li>
</ul>
<h3 id="4-hybrid-in-neo4j--selected">4. Hybrid in Neo4j ✅ Selected<a class="anchor" aria-label="Link to section" href="#4-hybrid-in-neo4j--selected">#</a></h3>
<ul>
<li><strong>Pros:</strong> one storage layer, one transaction model, one operational footprint. Both index types are first-class in Neo4j 5.</li>
<li><strong>Cons:</strong> Neo4j's vector indexes are newer than dedicated vector DBs; tuning is up to us.</li>
</ul>
<h2 id="rationale">Rationale<a class="anchor" aria-label="Link to section" href="#rationale">#</a></h2>
<p>The catalog is small enough (target: tens of thousands of entities) that there's no operational case for a separate search service. Fusing both signals in one query keeps the architecture simple. The discounts and penalties were tuned empirically against the Star Wars fixture and the dogfood catalog — they're constants and live in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>core/services/search_scoring.py</span></span></code></span> so they're easy to revisit.</p>
<p>The DSL is intentionally lax (unknown operators silently fall through to plain text). That's a power-user comfort vs. error-feedback trade-off. The expected user is technical and forgiving.</p>
<h2 id="consequences">Consequences<a class="anchor" aria-label="Link to section" href="#consequences">#</a></h2>
<h3 id="positive">Positive<a class="anchor" aria-label="Link to section" href="#positive">#</a></h3>
<ul>
<li>One database, one set of indexes, one query path.</li>
<li>Hybrid ranking handles both literal and semantic queries well.</li>
<li>Scoring constants are pure data — easy to tune, easy to test (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>tests/unit/test_search_scoring.py</span></span></code></span>).</li>
<li>Schema nodes are searchable individually but don't drown parent assets.</li>
</ul>
<h3 id="negative">Negative<a class="anchor" aria-label="Link to section" href="#negative">#</a></h3>
<ul>
<li>Embedding model adds memory pressure (~300 MB resident).</li>
<li>First search after API start is slow until the model is loaded.</li>
<li>Tuning constants is empirical; no automated relevance test.</li>
<li>Neo4j vector indexes are newer; behaviour at scale (millions of entities) is less proven than dedicated vector DBs.</li>
</ul>
<h3 id="mitigations">Mitigations<a class="anchor" aria-label="Link to section" href="#mitigations">#</a></h3>
<ul>
<li>Pre-warm the model at boot if needed.</li>
<li>Move to a dedicated vector store (Qdrant or similar) only when catalog size or query volume justifies the operational cost.</li>
</ul>
