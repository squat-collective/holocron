14:T42aa,<h1 id="adr-006-plugin-sdk--entry-point-discovery">ADR-006: Plugin SDK + entry-point discovery<a class="anchor" aria-label="Link to section" href="#adr-006-plugin-sdk--entry-point-discovery">#</a></h1>
<p><strong>Date:</strong> 2026-04-26
<strong>Status:</strong> Accepted (supersedes <a href="/docs/architecture/adr/003-reader-plugin-architecture">ADR-003</a>)
<strong>Deciders:</strong> Tom</p>
<h2 id="context">Context<a class="anchor" aria-label="Link to section" href="#context">#</a></h2>
<p>The MVP-era "reader" system (<a href="/docs/architecture/adr/003-reader-plugin-architecture">ADR-003</a>) had two assumptions that didn't survive contact with real use:</p>
<ol>
<li>
<p><strong>"Suggest don't write."</strong> Readers returned <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>Suggestion</span></span></code></span> objects; humans turned them into entities through a separate API call. Every connector ended up reimplementing the suggest→create plumbing, and the human-in-the-loop step was a friction point for batch imports — most users wanted "scan and ingest". The audit trail was already provided by <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>verified: false</span></span></code></span> + <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>discovered_by</span></span></code></span>, so the additional approval step was redundant.</p>
</li>
<li>
<p><strong>Reader-only.</strong> Real workflows needed exporters (catalog → file), audit plugins (catalog → findings), and detector plugins (catalog → suggestions for review). Naming everything a "reader" was misleading.</p>
</li>
</ol>
<p>We also needed a way for third parties to author plugins without depending on the whole API package (which pulls in FastAPI, Neo4j, embeddings, ...).</p>
<h2 id="decision">Decision<a class="anchor" aria-label="Link to section" href="#decision">#</a></h2>
<p>Replace readers with a unified <strong>plugin</strong> system:</p>
<ol>
<li><strong>Public SDK package</strong> (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>holocron-plugin-sdk</span></span></code></span>) with the manifest, input spec, result types, runtime context, and Protocol that every plugin satisfies. The API package depends on the SDK; plugins depend only on the SDK.</li>
<li><strong>Entry-point discovery.</strong> Plugins register via the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>holocron.plugins</span></span></code></span> entry-point group in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>pyproject.toml</span></span></code></span>. The API discovers them at startup (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>importlib.metadata.entry_points(group="holocron.plugins")</span></span></code></span>) — no filesystem scan, no manual registry.</li>
<li><strong>In-process invocation.</strong> <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>POST /api/v1/plugins/{slug}/run</span></span></code></span> parses multipart inputs against the manifest, builds a <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>PluginContext</span></span></code></span> carrying the API's service layer (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>asset_service</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>actor_service</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>relation_service</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>rule_service</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>event_service</span></span></code></span>), and awaits <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>plugin.run(ctx, inputs)</span></span></code></span>.</li>
<li><strong>Two capability families.</strong> <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>IMPORT</span></span></code></span> returns <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>SummaryResult</span></span></code></span> (JSON); <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>EXPORT</span></span></code></span> returns <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>DownloadResult</span></span></code></span> (streamed file). Audit and detector plugins fit naturally into one of these.</li>
<li><strong>CLI bundled with the SDK.</strong> <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>holocron-plugin list / show / run</span></span></code></span> for terminal-driven invocation, automation, and CI.</li>
<li><strong>Drop "suggest don't write".</strong> Plugins write directly through services. The audit trail is <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>verified: false</span></span></code></span> + <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>discovered_by: "&#x3C;slug>@&#x3C;version>"</span></span></code></span>, surfaced in the UI as a review queue.</li>
</ol>
<h2 id="options-considered">Options considered<a class="anchor" aria-label="Link to section" href="#options-considered">#</a></h2>
<h3 id="1-keep-the-reader-system-add-exporters-separately">1. Keep the reader system, add exporters separately<a class="anchor" aria-label="Link to section" href="#1-keep-the-reader-system-add-exporters-separately">#</a></h3>
<ul>
<li><strong>Pros:</strong> less churn.</li>
<li><strong>Cons:</strong> parallel plumbing for two APIs; readers and exporters end up duplicating manifest/discovery logic.</li>
</ul>
<h3 id="2-one-generic-system-but-configured-by-manifest-files-yaml">2. One generic system, but configured by manifest files (YAML)<a class="anchor" aria-label="Link to section" href="#2-one-generic-system-but-configured-by-manifest-files-yaml">#</a></h3>
<ul>
<li><strong>Pros:</strong> language-agnostic.</li>
<li><strong>Cons:</strong> extra layer to maintain, harder to test, no help from the type system. Python entry points are already a solved problem.</li>
</ul>
<h3 id="3-out-of-process-plugins-subprocess">3. Out-of-process plugins (subprocess)<a class="anchor" aria-label="Link to section" href="#3-out-of-process-plugins-subprocess">#</a></h3>
<ul>
<li><strong>Pros:</strong> isolation; bad plugin can't crash the API.</li>
<li><strong>Cons:</strong> IPC overhead, harder to debug, harder to share types. Premature for the current scale; revisit if/when plugins become untrusted.</li>
</ul>
<h3 id="4-generic-plugin-system-with-entry-point-discovery--public-sdk--selected">4. Generic plugin system with entry-point discovery + public SDK ✅ Selected<a class="anchor" aria-label="Link to section" href="#4-generic-plugin-system-with-entry-point-discovery--public-sdk--selected">#</a></h3>
<ul>
<li><strong>Pros:</strong> single contract, single discovery mechanism, single result-type set; SDK lets third parties build without depending on the API; entry points are the standard Python plugin idiom.</li>
<li><strong>Cons:</strong> plugins share the API process (a bad plugin can affect API performance); discovery only happens at startup (restart needed to add/remove).</li>
</ul>
<h2 id="rationale">Rationale<a class="anchor" aria-label="Link to section" href="#rationale">#</a></h2>
<p>Entry points are how Python plugins work. Reusing the standard mechanism means tooling (introspection, packaging, documentation) all just works. The SDK split is the key piece that enables third-party plugin authoring without dragging in FastAPI / Neo4j as dependencies.</p>
<p>Dropping "suggest don't write" reflects what users actually do: import everything, then sweep through the unverified set in the UI. The original design front-loaded the cost of approval before any data was in the catalog; the new design defers it, which matches how data teams actually work.</p>
<h2 id="consequences">Consequences<a class="anchor" aria-label="Link to section" href="#consequences">#</a></h2>
<h3 id="positive">Positive<a class="anchor" aria-label="Link to section" href="#positive">#</a></h3>
<ul>
<li>One mental model, one API surface for connectors, exporters, audit, detection.</li>
<li>Third parties can write plugins without depending on the API package.</li>
<li>The CLI gives automation parity with the UI.</li>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>PluginContext</span></span></code></span> makes plugins as powerful as built-in services — no awkward intermediate APIs.</li>
</ul>
<h3 id="negative">Negative<a class="anchor" aria-label="Link to section" href="#negative">#</a></h3>
<ul>
<li>Plugins share process memory with the API (a bad plugin can affect API performance).</li>
<li>Discovery is at startup; adding/removing a plugin requires an API restart.</li>
<li>No persistent job queue — plugin runs are synchronous for the request lifetime.</li>
</ul>
<h3 id="mitigations">Mitigations<a class="anchor" aria-label="Link to section" href="#mitigations">#</a></h3>
<ul>
<li>Treat plugins as first-party code; lint/typecheck/test in CI.</li>
<li>Document the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>verified: false</span></span></code></span> + <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>discovered_by</span></span></code></span> audit pattern as the standard.</li>
<li>If async/long-running plugins become common, add a job queue separately (don't change the synchronous contract).</li>
</ul>
