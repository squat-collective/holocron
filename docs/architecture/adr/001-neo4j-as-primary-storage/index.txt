14:T20a5,<h1 id="adr-001-neo4j-as-primary-storage">ADR-001: Neo4j as Primary Storage<a class="anchor" aria-label="Link to section" href="#adr-001-neo4j-as-primary-storage">#</a></h1>
<p><strong>Date:</strong> 2026-01-27
<strong>Status:</strong> Accepted
<strong>Deciders:</strong> Tom</p>
<h2 id="context">Context<a class="anchor" aria-label="Link to section" href="#context">#</a></h2>
<p>Holocron needs to store data assets, actors (people/groups), and their relationships. The core value proposition is <strong>visibility</strong> — understanding what data exists and how it connects.</p>
<p>Key requirements:</p>
<ul>
<li>Store entities with flexible metadata</li>
<li>Track typed relationships between entities</li>
<li>Query lineage (upstream/downstream traversal)</li>
<li>Answer "what depends on X?" questions efficiently</li>
</ul>
<h2 id="decision">Decision<a class="anchor" aria-label="Link to section" href="#decision">#</a></h2>
<p><strong>Use Neo4j as the primary (and only) database.</strong></p>
<h2 id="options-considered">Options Considered<a class="anchor" aria-label="Link to section" href="#options-considered">#</a></h2>
<h3 id="1-postgresql-relational">1. PostgreSQL (Relational)<a class="anchor" aria-label="Link to section" href="#1-postgresql-relational">#</a></h3>
<p><strong>Pros:</strong></p>
<ul>
<li>Familiar, battle-tested</li>
<li>Strong ACID guarantees</li>
<li>Good tooling ecosystem</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>Recursive CTEs for lineage queries are complex and slow</li>
<li>Schema migrations for new entity types</li>
<li>JOIN-heavy queries for relationship traversal</li>
</ul>
<h3 id="2-mongodb-document">2. MongoDB (Document)<a class="anchor" aria-label="Link to section" href="#2-mongodb-document">#</a></h3>
<p><strong>Pros:</strong></p>
<ul>
<li>Flexible schema</li>
<li>Easy to start</li>
<li>Good for metadata blobs</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>No native graph traversal</li>
<li>Would need application-level relationship handling</li>
<li>Lineage queries become complex</li>
</ul>
<h3 id="3-neo4j-graph--selected">3. Neo4j (Graph) ✅ Selected<a class="anchor" aria-label="Link to section" href="#3-neo4j-graph--selected">#</a></h3>
<p><strong>Pros:</strong></p>
<ul>
<li>Native graph traversal (lineage in one query)</li>
<li>Cypher is expressive for relationship queries</li>
<li>Multi-label nodes fit our model perfectly</li>
<li>Built-in visualization (Neo4j Browser)</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>Less familiar than SQL</li>
<li>Smaller talent pool</li>
<li>Overkill if we don't use graph features</li>
</ul>
<h3 id="4-hybrid-postgres--neo4j">4. Hybrid (Postgres + Neo4j)<a class="anchor" aria-label="Link to section" href="#4-hybrid-postgres--neo4j">#</a></h3>
<p><strong>Pros:</strong></p>
<ul>
<li>Best of both worlds</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>Sync complexity</li>
<li>Two databases to maintain</li>
<li>Overkill for MVP</li>
</ul>
<h2 id="rationale">Rationale<a class="anchor" aria-label="Link to section" href="#rationale">#</a></h2>
<p>The <strong>core value</strong> of Holocron is understanding relationships:</p>
<ul>
<li>"What feeds into this report?"</li>
<li>"What breaks if I change this dataset?"</li>
<li>"Who owns the things that depend on X?"</li>
</ul>
<p>These are <strong>graph traversal queries</strong>. In a relational DB, they require recursive CTEs or multiple round-trips. In Neo4j, they're single Cypher queries:</p>

<p>Given that <strong>relationship traversal is the core feature</strong>, Neo4j is the right tool.</p>
<h2 id="consequences">Consequences<a class="anchor" aria-label="Link to section" href="#consequences">#</a></h2>
<h3 id="positive">Positive<a class="anchor" aria-label="Link to section" href="#positive">#</a></h3>
<ul>
<li>Lineage and impact queries are simple and fast</li>
<li>Data model maps naturally to graph</li>
<li>Can visualize relationships in Neo4j Browser</li>
<li>Cypher is readable for non-engineers</li>
</ul>
<h3 id="negative">Negative<a class="anchor" aria-label="Link to section" href="#negative">#</a></h3>
<ul>
<li>Team needs to learn Cypher</li>
<li>Fewer hosting options than Postgres</li>
<li>May need Postgres later for other features (auth, audit logs)</li>
</ul>
<h3 id="mitigations">Mitigations<a class="anchor" aria-label="Link to section" href="#mitigations">#</a></h3>
<ul>
<li>Document common Cypher patterns</li>
<li>Use Neo4j's official Python driver (well-supported)</li>
<li>Keep the door open for adding Postgres later if needed</li>
</ul>
