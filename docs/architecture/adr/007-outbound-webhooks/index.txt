14:T32fb,<h1 id="adr-007-outbound-webhooks-hmac-fire-and-forget">ADR-007: Outbound webhooks (HMAC, fire-and-forget)<a class="anchor" aria-label="Link to section" href="#adr-007-outbound-webhooks-hmac-fire-and-forget">#</a></h1>
<p><strong>Date:</strong> 2026-04-26
<strong>Status:</strong> Accepted
<strong>Deciders:</strong> Tom</p>
<h2 id="context">Context<a class="anchor" aria-label="Link to section" href="#context">#</a></h2>
<p>Every mutation in Holocron logs an <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:Event</span></span></code></span> node. External systems (alerting, analytics, governance pipelines) need to react to those events without polling. We need:</p>
<ul>
<li>A way for receivers to subscribe by event topic.</li>
<li>Authenticity (the receiver should be able to verify the request came from Holocron).</li>
<li>Reasonable failure handling (don't melt the API if a receiver is down).</li>
<li>A test path so receivers can validate their integration without waiting for real events.</li>
</ul>
<h2 id="decision">Decision<a class="anchor" aria-label="Link to section" href="#decision">#</a></h2>
<p>Implement outbound webhooks as a first-class entity with HMAC-SHA256 signing and fire-and-forget dispatch:</p>
<ol>
<li><strong><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:Webhook</span></span></code></span> nodes</strong> with <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>url</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>events</span></span></code></span> (list of <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>&#x3C;entity>.&#x3C;action></span></span></code></span> topics or <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>*</span></span></code></span>), <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>secret</span></span></code></span> (HMAC key), <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>description</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>disabled</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>failure_count</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>last_error</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>last_fired_at</span></span></code></span>.</li>
<li><strong>CRUD via <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>/api/v1/webhooks</span></span></code></span></strong> — create returns the secret <strong>once</strong>; subsequent reads never expose it.</li>
<li><strong>Fan-out at event time.</strong> When <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>event_service.log()</span></span></code></span> writes an <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:Event</span></span></code></span>, it triggers <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>webhook_dispatcher.dispatch_event(event)</span></span></code></span> which iterates matching subscribers and dispatches each in an <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>asyncio.create_task()</span></span></code></span>. The mutation request returns immediately.</li>
<li><strong>HMAC-SHA256 over the raw body</strong>, sent in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>X-Holocron-Signature: sha256=&#x3C;hex></span></span></code></span>. Topic in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>X-Holocron-Topic</span></span></code></span>, event UID in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>X-Holocron-Event-Uid</span></span></code></span>. <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>User-Agent: holocron-webhooks/0.1</span></span></code></span>.</li>
<li><strong>Bounded concurrency.</strong> Process-wide <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>asyncio.Semaphore(100)</span></span></code></span> caps in-flight deliveries.</li>
<li><strong>Auto-disable.</strong> After 10 consecutive failures the webhook is disabled. Re-enable via <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>PUT {"disabled": false}</span></span></code></span>, which also resets the counter.</li>
<li><strong>No persistent retry queue (v0.1).</strong> A failed delivery is not retried beyond the immediate attempt. Receivers handle eventual consistency by reconciling against <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>GET /events</span></span></code></span>.</li>
<li><strong>Test endpoint.</strong> <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>POST /webhooks/{uid}/test</span></span></code></span> fires a synthetic event with <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata.test=true</span></span></code></span> and returns <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>{"delivered": true|false}</span></span></code></span>.</li>
</ol>
<h2 id="options-considered">Options considered<a class="anchor" aria-label="Link to section" href="#options-considered">#</a></h2>
<h3 id="1-polling">1. Polling<a class="anchor" aria-label="Link to section" href="#1-polling">#</a></h3>
<ul>
<li><strong>Pros:</strong> trivial — receivers GET <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>/events?since=...</span></span></code></span>.</li>
<li><strong>Cons:</strong> chatty, latency floor = poll interval, can't push to slack/pagerduty cleanly.</li>
</ul>
<h3 id="2-webhooks-with-persistent-retry-queue">2. Webhooks with persistent retry queue<a class="anchor" aria-label="Link to section" href="#2-webhooks-with-persistent-retry-queue">#</a></h3>
<ul>
<li><strong>Pros:</strong> robust against transient receiver failures.</li>
<li><strong>Cons:</strong> introduces a queue (Redis / Postgres / RabbitMQ) — extra service to operate. Premature for the current scale.</li>
</ul>
<h3 id="3-push-to-a-message-broker-kafka--nats">3. Push to a message broker (Kafka / NATS)<a class="anchor" aria-label="Link to section" href="#3-push-to-a-message-broker-kafka--nats">#</a></h3>
<ul>
<li><strong>Pros:</strong> scales horizontally, fan-out is the broker's problem.</li>
<li><strong>Cons:</strong> receivers must speak the broker protocol, extra infrastructure, very different operational model.</li>
</ul>
<h3 id="4-outbound-http-webhooks-with-hmac-fire-and-forget--selected">4. Outbound HTTP webhooks with HMAC, fire-and-forget ✅ Selected<a class="anchor" aria-label="Link to section" href="#4-outbound-http-webhooks-with-hmac-fire-and-forget--selected">#</a></h3>
<ul>
<li><strong>Pros:</strong> zero extra infrastructure; receivers only need an HTTP endpoint and a shared secret; HMAC gives authenticity without TLS-only solutions.</li>
<li><strong>Cons:</strong> transient failures lose events (mitigated by <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>/events</span></span></code></span> reconciliation); secrets stored plaintext (mitigated by treating the DB as sensitive).</li>
</ul>
<h2 id="rationale">Rationale<a class="anchor" aria-label="Link to section" href="#rationale">#</a></h2>
<p>Webhooks are the lowest-friction option for receivers — every alerting / governance / orchestration tool understands a signed POST. Fire-and-forget keeps the API responsive: we never block a write on a slow receiver. HMAC over the raw body is the standard pattern (Stripe, GitHub, ...), familiar to anyone integrating webhooks before.</p>
<p>A persistent retry queue is the obvious next step but adds a service. It can be added later without changing the wire contract — receivers wouldn't notice. For now, "if you missed an event, replay from <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>/events</span></span></code></span>" is the documented path.</p>
<p>The 10-failure auto-disable threshold protects the API from wasting effort on permanently-broken receivers. It's high enough to ride out transient outages and low enough that a typoed URL stops sending within a minute or so of meaningful traffic.</p>
<h2 id="consequences">Consequences<a class="anchor" aria-label="Link to section" href="#consequences">#</a></h2>
<h3 id="positive">Positive<a class="anchor" aria-label="Link to section" href="#positive">#</a></h3>
<ul>
<li>Zero extra infrastructure.</li>
<li>Standard HMAC pattern, easy to integrate.</li>
<li>Mutations never block on receivers.</li>
<li>Bad receivers self-disable.</li>
</ul>
<h3 id="negative">Negative<a class="anchor" aria-label="Link to section" href="#negative">#</a></h3>
<ul>
<li>Failed deliveries aren't retried (in v0.1).</li>
<li>Secrets are stored plaintext in Neo4j (the dispatcher needs them to sign).</li>
<li>No delivery ordering guarantees across receivers.</li>
<li>The semaphore is per-process; multiple workers don't share a global concurrency cap.</li>
</ul>
<h3 id="mitigations">Mitigations<a class="anchor" aria-label="Link to section" href="#mitigations">#</a></h3>
<ul>
<li>Document <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>/events</span></span></code></span> as the reconciliation path.</li>
<li>Treat the Neo4j database as sensitive (it always was — owners and metadata live there too).</li>
<li>If at-least-once delivery becomes a requirement, add a persistent queue without changing the wire contract.</li>
</ul>
