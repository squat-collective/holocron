14:T3652,<h1 id="adr-008-schema-projection--containerfield-nodes">ADR-008: Schema projection — Container/Field nodes<a class="anchor" aria-label="Link to section" href="#adr-008-schema-projection--containerfield-nodes">#</a></h1>
<p><strong>Date:</strong> 2026-04-25
<strong>Status:</strong> Accepted
<strong>Deciders:</strong> Tom</p>
<h2 id="context">Context<a class="anchor" aria-label="Link to section" href="#context">#</a></h2>
<p>Connectors discover datasets <em>with schema</em> — a CSV has columns, an Excel sheet has tables and columns, a Postgres table has columns with types, a Power BI report references tables and fields. The schema lives naturally in the asset's <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata.schema</span></span></code></span> field as a nested JSON tree (containers → fields).</p>
<p>But putting it only in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata</span></span></code></span> makes it invisible to:</p>
<ul>
<li><strong>Search.</strong> Users want to find "the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>email</span></span></code></span> field of the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>customers</span></span></code></span> dataset", not just "the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>customers</span></span></code></span> dataset that happens to mention email somewhere".</li>
<li><strong>Governance.</strong> Rules apply to specific fields (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>pii: true</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>must_be_masked</span></span></code></span>); we need to attach <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:APPLIES_TO</span></span></code></span> edges scoped to a <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>field_path</span></span></code></span>.</li>
<li><strong>Lineage.</strong> Column-level lineage between datasets requires column nodes to anchor edges to.</li>
</ul>
<p>Storing schema as opaque JSON gives us none of those.</p>
<h2 id="decision">Decision<a class="anchor" aria-label="Link to section" href="#decision">#</a></h2>
<p>Materialise the JSON schema tree as real Neo4j nodes:</p>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="plaintext" data-theme="github-dark-dimmed github-light"><code data-language="plaintext" data-theme="github-dark-dimmed github-light" style="display: grid;"><span data-line=""><span>:Asset --(:CONTAINS)--> :Container --(:CONTAINS)--> :Field</span></span></code></pre></figure>
<p>Triggered automatically whenever an Asset's <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata.schema</span></span></code></span> is written. Implemented as a <strong>full rebuild</strong> of that asset's subtree:</p>
<ol>
<li>Tear down: detach-delete every <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:Container</span></span></code></span> and <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:Field</span></span></code></span> reachable from the asset.</li>
<li>Materialise: walk the new schema JSON, creating containers and fields with denormalised <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>asset_uid</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>asset_name</span></span></code></span>, slash-joined <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>path</span></span></code></span> (e.g. <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>customers/email</span></span></code></span>).</li>
<li>Embed: the embedding service produces a 384-dim vector for each new container and field, populating the same vector + fulltext indexes used by other entity kinds.</li>
</ol>
<p>Container and Field nodes are <strong>derived state</strong> — they're not directly created or edited via the API. Editing them happens through the parent asset's <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata.schema</span></span></code></span>.</p>
<p>Source: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>packages/api/src/holocron/core/services/asset_schema_projection.py</span></span></code></span>.</p>
<h2 id="options-considered">Options considered<a class="anchor" aria-label="Link to section" href="#options-considered">#</a></h2>
<h3 id="1-keep-schema-as-json-in-metadata">1. Keep schema as JSON in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata</span></span></code></span><a class="anchor" aria-label="Link to section" href="#1-keep-schema-as-json-in-metadata">#</a></h3>
<ul>
<li><strong>Pros:</strong> simple, no projection step.</li>
<li><strong>Cons:</strong> invisible to search and governance. Users have to write Cypher to find a column.</li>
</ul>
<h3 id="2-materialise-lazily-on-first-read">2. Materialise lazily on first read<a class="anchor" aria-label="Link to section" href="#2-materialise-lazily-on-first-read">#</a></h3>
<ul>
<li><strong>Pros:</strong> writes stay cheap.</li>
<li><strong>Cons:</strong> the first read after a write is slow and unpredictable. Search needs the indexes populated <em>now</em>, not on first query. Hard to keep consistent.</li>
</ul>
<h3 id="3-store-as-a-child-neo4j-sub-api-post-assetsuidcolumns-">3. Store as a child Neo4j sub-API (POST <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>/assets/{uid}/columns</span></span></code></span>, ...)<a class="anchor" aria-label="Link to section" href="#3-store-as-a-child-neo4j-sub-api-post-assetsuidcolumns-">#</a></h3>
<ul>
<li><strong>Pros:</strong> explicit lifecycle.</li>
<li><strong>Cons:</strong> users now have two APIs to keep in sync; connectors have to make N+M calls per asset; the JSON in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata</span></span></code></span> becomes ambiguous (which is canonical?).</li>
</ul>
<h3 id="4-materialise-on-every-write--selected">4. Materialise on every write ✅ Selected<a class="anchor" aria-label="Link to section" href="#4-materialise-on-every-write--selected">#</a></h3>
<ul>
<li><strong>Pros:</strong> the JSON in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata</span></span></code></span> stays canonical (one source of truth); the graph stays consistent; search and governance work without special-casing schema; connectors only need to set <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata.schema</span></span></code></span>.</li>
<li><strong>Cons:</strong> every asset write rebuilds the subtree even if the schema didn't change. Cost is bounded by per-asset schema size.</li>
</ul>
<h3 id="5-diff-based-incremental-rebuild">5. Diff-based incremental rebuild<a class="anchor" aria-label="Link to section" href="#5-diff-based-incremental-rebuild">#</a></h3>
<ul>
<li><strong>Pros:</strong> cheaper at scale.</li>
<li><strong>Cons:</strong> more code, more failure modes (drifting state if the diff is wrong). Premature for current scale.</li>
</ul>
<h2 id="rationale">Rationale<a class="anchor" aria-label="Link to section" href="#rationale">#</a></h2>
<p>JSON in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata</span></span></code></span> is what connectors naturally produce, and it's the easiest authoring path for hand-editing. Materialising on write keeps the JSON canonical (it's the input; the nodes are the output) without forcing a second authoring API. The full rebuild is dumb but correct — diff-based projection can come later if writes become a bottleneck.</p>
<p>The denormalisation of <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>asset_uid</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>asset_name</span></span></code></span>, and <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>path</span></span></code></span> onto every Container/Field means search results can render "the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>email</span></span></code></span> field of the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>customers</span></span></code></span> table" without a join back to the parent. This trades a little storage for a lot of read simplicity, which is worth it for the catalog scale we target.</p>
<h2 id="consequences">Consequences<a class="anchor" aria-label="Link to section" href="#consequences">#</a></h2>
<h3 id="positive">Positive<a class="anchor" aria-label="Link to section" href="#positive">#</a></h3>
<ul>
<li>Schema is a first-class part of the graph: search, governance, lineage all see fields and containers.</li>
<li>The JSON in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata</span></span></code></span> stays canonical — no two-source-of-truth problem.</li>
<li>Connectors are unchanged: just set <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata.schema</span></span></code></span>.</li>
<li>Search hits gain <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>kind: "container"</span></span></code></span> and <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>kind: "field"</span></span></code></span> for free.</li>
<li>Rules can attach to specific fields via <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>field_path</span></span></code></span> on <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:APPLIES_TO</span></span></code></span>.</li>
</ul>
<h3 id="negative">Negative<a class="anchor" aria-label="Link to section" href="#negative">#</a></h3>
<ul>
<li>Every asset write rebuilds the subtree, even if the schema didn't change.</li>
<li>At very deep schemas (>1000 fields per asset) the rebuild + embedding cost is noticeable.</li>
<li>Container/Field UIDs are stable per asset+path but not portable (if the asset is renamed, paths change).</li>
</ul>
<h3 id="mitigations">Mitigations<a class="anchor" aria-label="Link to section" href="#mitigations">#</a></h3>
<ul>
<li>Skip the rebuild if the schema field didn't change in the update payload (cheap diff at the route layer — TODO).</li>
<li>Move embedding to a batch job at scale (the same way the graph layout is currently cached at startup).</li>
</ul>
