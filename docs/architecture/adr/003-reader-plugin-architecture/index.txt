14:T451c,<h1 id="adr-003-reader-plugin-architecture">ADR-003: Reader Plugin Architecture<a class="anchor" aria-label="Link to section" href="#adr-003-reader-plugin-architecture">#</a></h1>
<p><strong>Date:</strong> 2026-01-27
<strong>Status:</strong> ⚠️ <strong>Superseded by <a href="/docs/architecture/adr/006-plugin-sdk-entry-points">ADR-006</a></strong> (2026-04-26)
<strong>Deciders:</strong> Tom</p>
<blockquote>
<p>The "reader" naming and the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>/readers/{name}/scan</span></span></code></span> API have been replaced by
the generic plugin system: a public <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>holocron-plugin-sdk</span></span></code></span> package, entry-point
discovery via the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>holocron.plugins</span></span></code></span> group, in-process invocation through
<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>POST /api/v1/plugins/{slug}/run</span></span></code></span>, and three capability families
(connectors, exporters, audit/extension plugins). The "suggest don't write"
model was dropped — plugins now have full service access via <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>PluginContext</span></span></code></span>.
See <a href="/docs/architecture/adr/006-plugin-sdk-entry-points">ADR-006</a> for the rationale.</p>
</blockquote>
<h2 id="context">Context<a class="anchor" aria-label="Link to section" href="#context">#</a></h2>
<p>Holocron needs "readers" — components that scan external sources (Excel files, databases, etc.) and suggest metadata about data assets. This is how Holocron learns about existing data without requiring 100% manual entry.</p>
<p>Key requirements:</p>
<ul>
<li>Readers should be extensible (add new source types)</li>
<li>Readers output <strong>suggestions</strong>, not direct writes (human approval required)</li>
<li>Simple to implement new readers</li>
<li>Must work in containerized environment</li>
</ul>
<h2 id="decision">Decision<a class="anchor" aria-label="Link to section" href="#decision">#</a></h2>
<p><strong>Implement readers as in-process Python plugins that return suggestions via API.</strong></p>
<h3 id="reader-behavior">Reader Behavior<a class="anchor" aria-label="Link to section" href="#reader-behavior">#</a></h3>
<ol>
<li>Readers are triggered via API: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>POST /api/v1/readers/{name}/scan</span></span></code></span></li>
<li>Readers receive a source path/URL and options</li>
<li>Readers return suggestions (ephemeral, not persisted)</li>
<li>Humans review suggestions and create assets via separate API calls</li>
</ol>
<h3 id="plugin-structure">Plugin Structure<a class="anchor" aria-label="Link to section" href="#plugin-structure">#</a></h3>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="plaintext" data-theme="github-dark-dimmed github-light"><code data-language="plaintext" data-theme="github-dark-dimmed github-light" style="display: grid;"><span data-line=""><span>plugins/</span></span>
<span data-line=""><span>└── excel_reader/</span></span>
<span data-line=""><span>    ├── __init__.py</span></span>
<span data-line=""><span>    └── reader.py      # Contains class extending BaseReader</span></span></code></pre></figure>
<h2 id="options-considered">Options Considered<a class="anchor" aria-label="Link to section" href="#options-considered">#</a></h2>
<h3 id="1-in-process-python-modules--selected">1. In-Process Python Modules ✅ Selected<a class="anchor" aria-label="Link to section" href="#1-in-process-python-modules--selected">#</a></h3>
<p><strong>Pros:</strong></p>
<ul>
<li>Simplest to implement</li>
<li>No IPC overhead</li>
<li>Easy debugging</li>
<li>Shared types with core</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>Bad reader can crash the app</li>
<li>All readers must be Python</li>
<li>Memory shared with main process</li>
</ul>
<h3 id="2-subprocess-exec">2. Subprocess (exec)<a class="anchor" aria-label="Link to section" href="#2-subprocess-exec">#</a></h3>
<p><strong>Pros:</strong></p>
<ul>
<li>Process isolation</li>
<li>Can be any language</li>
<li>Crash isolation</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>IPC complexity (stdin/stdout JSON)</li>
<li>Harder to debug</li>
<li>Dependency management per reader</li>
</ul>
<h3 id="3-container-per-reader">3. Container per Reader<a class="anchor" aria-label="Link to section" href="#3-container-per-reader">#</a></h3>
<p><strong>Pros:</strong></p>
<ul>
<li>Maximum isolation</li>
<li>Independent scaling</li>
<li>Any language/runtime</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>Complex orchestration</li>
<li>Slow startup</li>
<li>Overkill for MVP</li>
</ul>
<h3 id="4-external-services-webhooks">4. External Services (webhooks)<a class="anchor" aria-label="Link to section" href="#4-external-services-webhooks">#</a></h3>
<p><strong>Pros:</strong></p>
<ul>
<li>Fully decoupled</li>
<li>Can run anywhere</li>
</ul>
<p><strong>Cons:</strong></p>
<ul>
<li>Network complexity</li>
<li>Auth/security concerns</li>
<li>Deployment overhead</li>
</ul>
<h2 id="rationale">Rationale<a class="anchor" aria-label="Link to section" href="#rationale">#</a></h2>
<p>For MVP, <strong>simplicity wins</strong>. In-process plugins:</p>
<ul>
<li>Are trivial to implement (just Python classes)</li>
<li>Share Pydantic models with the core</li>
<li>Can be unit tested easily</li>
<li>Don't require container orchestration</li>
</ul>
<p>The design <strong>doesn't prevent</strong> moving to subprocess or containers later. The <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>BaseReader</span></span></code></span> interface is the contract — implementation can change.</p>
<h3 id="suggestion-model-not-direct-writes">Suggestion Model (Not Direct Writes)<a class="anchor" aria-label="Link to section" href="#suggestion-model-not-direct-writes">#</a></h3>
<p>Critical design choice: readers <strong>suggest</strong>, they don't create.</p>

<span data-line=""> </span>
<span data-line=""><span style="--shiki-dark:#768390;--shiki-light:#6A737D"># Human reviews and creates</span></span>
<span data-line=""><span style="--shiki-dark:#6CB6FF;--shiki-light:#005CC5">POST</span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49"> /</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">assets {</span><span style="--shiki-dark:#96D0FF;--shiki-light:#032F62">"type"</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">: </span><span style="--shiki-dark:#96D0FF;--shiki-light:#032F62">"dataset"</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">, </span><span style="--shiki-dark:#96D0FF;--shiki-light:#032F62">"name"</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">: </span><span style="--shiki-dark:#96D0FF;--shiki-light:#032F62">"Sheet1"</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">, </span><span style="--shiki-dark:#6CB6FF;--shiki-light:#005CC5">...</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E">}</span></span></code></pre></figure>
<p>Why:</p>
<ul>
<li><strong>Human stays in control</strong> — readers can be wrong</li>
<li><strong>Auditability</strong> — humans are accountable for what's in Holocron</li>
<li><strong>Simplicity</strong> — readers don't need write access to DB</li>
<li><strong>KISS</strong> — no approval workflow state machine needed</li>
</ul>
<h2 id="consequences">Consequences<a class="anchor" aria-label="Link to section" href="#consequences">#</a></h2>
<h3 id="positive">Positive<a class="anchor" aria-label="Link to section" href="#positive">#</a></h3>
<ul>
<li>Easy to write new readers (just extend BaseReader)</li>
<li>No infrastructure complexity</li>
<li>Suggestions are simple API responses</li>
<li>Clear separation: readers suggest, humans decide</li>
</ul>
<h3 id="negative">Negative<a class="anchor" aria-label="Link to section" href="#negative">#</a></h3>
<ul>
<li>Readers share process memory</li>
<li>A bad reader can affect API performance</li>
<li>All readers must be Python</li>
</ul>
<h3 id="mitigations">Mitigations<a class="anchor" aria-label="Link to section" href="#mitigations">#</a></h3>
<ul>
<li>Timeouts on reader scan operations</li>
<li>Input validation before passing to readers</li>
<li>Consider process isolation if readers become untrusted</li>
</ul>
