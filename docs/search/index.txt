14:T5c61,<h1 id="search">Search<a class="anchor" aria-label="Link to section" href="#search">#</a></h1>
<blockquote>
<p>Hybrid semantic + keyword search across every entity in the catalog, with a small DSL for power users.</p>
</blockquote>

<p>Results are interleaved by score, not grouped by kind. A perfectly named field will rank above a vaguely matching asset.</p>
<h2 id="how-scoring-works">How scoring works<a class="anchor" aria-label="Link to section" href="#how-scoring-works">#</a></h2>
<p>Every searchable kind has both:</p>
<ul>
<li>A <strong>vector index</strong> over a 384-dim embedding of <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>name + description</span></span></code></span> (BAAI/bge-small-en-v1.5, ONNX via fastembed, CPU-only).</li>
<li>A <strong>fulltext index</strong> (Lucene through Neo4j) over <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>name + description</span></span></code></span>.</li>
</ul>
<p>For each query, the API:</p>
<ol>
<li>Parses the query into the structured <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>ParsedQuery</span></span></code></span> (filters, kinds, phrases, negations, graph filters — see DSL below).</li>
<li>Resolves any graph filters (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>owner:</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>uses:</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>feeds:</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>rule_for:</span></span></code></span>) into UID sets via fulltext.</li>
<li>Fans out per-kind rankers: vector top-N + fulltext top-N.</li>
<li>Fuses scores with the <strong>probabilistic OR</strong> rule: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>combined = 1 − (1 − v) × (1 − fts)</span></span></code></span>.</li>
<li>Applies <strong>kind discounts</strong> so schema nodes don't drown out their parent assets:
<ul>
<li>container × <strong>0.85</strong></li>
<li>field × <strong>0.80</strong></li>
</ul>
</li>
<li>Penalises pure-vector hits when fulltext is dominant: if any FTS hit is above <strong>0.55</strong>, vector-only hits are scaled by <strong>0.55</strong>.</li>
<li>Sorts by combined score; ties break by kind order.</li>
</ol>
<p>Constants live in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>packages/api/src/holocron/core/services/search_scoring.py</span></span></code></span> and have unit tests in <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>packages/api/tests/unit/test_search_scoring.py</span></span></code></span>.</p>
<p>See <a href="/docs/architecture/adr/005-hybrid-search">ADR-005</a> for the rationale.</p>

<h2 id="forgiveness-rules">Forgiveness rules<a class="anchor" aria-label="Link to section" href="#forgiveness-rules">#</a></h2>
<p>The parser is intentionally lax:</p>
<ul>
<li>Unknown aliases fall back to plain text (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>xyz:foo</span></span></code></span> becomes a semantic search for <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>xyz:foo</span></span></code></span>).</li>
<li>Typos in operator names silently become semantic queries — they don't error out.</li>
<li>Empty <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>q</span></span></code></span> is allowed and returns the most recently updated entities.</li>
</ul>
<p>The trade-off is power-user comfort vs. error feedback. If you want strict validation, use the SDK's typed helpers (when they ship for search).</p>

<h2 id="performance-notes">Performance notes<a class="anchor" aria-label="Link to section" href="#performance-notes">#</a></h2>
<ul>
<li>Embedding model is loaded <strong>lazily</strong> on first search request (~1 s, ~300 MB resident). After that, queries are sub-100 ms typical.</li>
<li>Vector indexes are populated synchronously on entity write — first search after a bulk import will reflect new entities immediately.</li>
<li>Schema projection rebuilds on every asset update; large schemas (>1k fields) make writes slower. There's a TODO to move to a batch job.</li>
</ul>
<h2 id="known-gaps">Known gaps<a class="anchor" aria-label="Link to section" href="#known-gaps">#</a></h2>
<ul>
<li>No saved searches or query templates.</li>
<li>No relevance feedback (clicks aren't fed back into ranking).</li>
<li>No relations fetch from search hit (would need a follow-up call to <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>/relations</span></span></code></span>).</li>
<li>DSL aliases are hardcoded — not extensible without a code change.</li>
