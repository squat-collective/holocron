14:T780e,<h1 id="concepts">Concepts<a class="anchor" aria-label="Link to section" href="#concepts">#</a></h1>
<blockquote>
<p>The data model in one page.</p>
</blockquote>
<p>Holocron stores everything as a Neo4j graph. There are five top-level entity kinds, two derived schema kinds, six relationship types, and a small set of enums. Read this once and the rest of the docs make sense.</p>

<p>Containers and Fields aren't created by API consumers — they're materialised by the <strong>schema projection</strong> (see below) when an Asset's <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata.schema</span></span></code></span> is updated.</p>
<h3 id="multi-label-nodes">Multi-label nodes<a class="anchor" aria-label="Link to section" href="#multi-label-nodes">#</a></h3>
<p>Assets and Actors carry both a base label (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:Asset</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:Actor</span></span></code></span>) and a type-specific label (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:Dataset</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:Person</span></span></code></span>, ...). This is Neo4j's recommended pattern and lets queries scope at either level:</p>

<span data-line=""><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">MATCH</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> (d:Dataset)      </span><span style="--shiki-dark:#F47067;--shiki-light:#D73A49">RETURN</span><span style="--shiki-dark:#ADBAC7;--shiki-light:#24292E"> d       </span><span style="--shiki-dark:#768390;--shiki-light:#6A737D">// only datasets</span></span>

<h3 id="common-fields">Common fields<a class="anchor" aria-label="Link to section" href="#common-fields">#</a></h3>
<p>Every entity has:</p>
<ul>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>uid</span></span></code></span> (string, unique) — assigned at creation, never reused.</li>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>name</span></span></code></span> (string) — display label.</li>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>description</span></span></code></span> (string, optional).</li>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>verified</span></span></code></span> (bool) — was this confirmed by a human, or is it a plugin's guess?</li>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>discovered_by</span></span></code></span> (string, optional) — <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>&#x3C;plugin-slug>@&#x3C;version></span></span></code></span> if a plugin created it.</li>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>created_at</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>updated_at</span></span></code></span> (datetime).</li>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata</span></span></code></span> (JSON object) — anything that doesn't fit a typed field.</li>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>embedding</span></span></code></span> (384-dim float vector) — populated by the embedding service (BAAI/bge-small-en-v1.5).</li>

<h3 id="lineage-vocabulary">Lineage vocabulary<a class="anchor" aria-label="Link to section" href="#lineage-vocabulary">#</a></h3>
<p>Holocron uses <strong>only <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:FEEDS</span></span></code></span></strong> for lineage. There's no separate <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:PRODUCES</span></span></code></span> / <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:CONSUMES</span></span></code></span> / <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:DERIVED_FROM</span></span></code></span>. A process that loads a mart is just an <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>Asset(type=process)</span></span></code></span> with <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>feeds</span></span></code></span> going in (from sources) and <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>feeds</span></span></code></span> going out (to the mart):</p>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="plaintext" data-theme="github-dark-dimmed github-light"><code data-language="plaintext" data-theme="github-dark-dimmed github-light" style="display: grid;"><span data-line=""><span>sources (datasets) -[:FEEDS]→ ETL job (process) -[:FEEDS]→ data mart (dataset)</span></span></code></pre></figure>
<p>This keeps the graph asset-only and makes lineage walks trivial: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>MATCH (a)-[:FEEDS*]->(downstream)</span></span></code></span>.</p>

<h2 id="schema-projection">Schema projection<a class="anchor" aria-label="Link to section" href="#schema-projection">#</a></h2>
<p>When you write an Asset whose <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata.schema</span></span></code></span> describes containers and fields, the API materialises that JSON into real graph nodes:</p>
<figure data-rehype-pretty-code-figure=""><pre tabindex="0" data-language="plaintext" data-theme="github-dark-dimmed github-light"><code data-language="plaintext" data-theme="github-dark-dimmed github-light" style="display: grid;"><span data-line=""><span>:Asset --(:CONTAINS)--> :Container --(:CONTAINS)--> :Field</span></span></code></pre></figure>
<ul>
<li>Container/Field nodes carry a denormalised <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>asset_uid</span></span></code></span>, <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>asset_name</span></span></code></span>, and slash-joined <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>path</span></span></code></span> so search can show "the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>email</span></span></code></span> field of the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>customers</span></span></code></span> table".</li>
<li>Each gets its own embedding and lives in the vector + fulltext indexes — column-level search works out of the box.</li>
<li>The projection is <strong>rebuilt</strong> on every asset write: tear down the old subtree, materialise the new one. Simple and consistent; the cost is bounded by the per-asset schema size.</li>
</ul>
<p>Source: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>packages/api/src/holocron/core/services/asset_schema_projection.py</span></span></code></span>.</p>
<p>See <a href="/docs/architecture/adr/008-schema-projection">ADR-008</a>.</p>
<h2 id="audit-trail-events">Audit trail (Events)<a class="anchor" aria-label="Link to section" href="#audit-trail-events">#</a></h2>
<p>Every mutation logs an <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:Event</span></span></code></span> node. Events are queryable (<span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>GET /api/v1/events</span></span></code></span>) and they're the trigger for outbound webhooks — register a webhook for <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>asset.updated</span></span></code></span> and you'll get a signed POST every time anyone updates an asset.</p>
<p>There's no soft-delete: <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>DELETE /assets/{uid}</span></span></code></span> removes the node, but the <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>event</span></span></code></span> recording the deletion stays.</p>
<h2 id="identity--idempotency">Identity &#x26; idempotency<a class="anchor" aria-label="Link to section" href="#identity--idempotency">#</a></h2>
<p>Connectors typically derive UIDs from a deterministic hash of the source identity (file path, table FQN, ...). That makes re-scans <strong>upsert</strong> rather than create duplicates. The catalog UID is kept stable across scans even when the entity's properties change.</p>
<p>Examples (see each connector's README for the exact recipe):</p>
<ul>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>csv-connector</span></span></code></span> — <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>sha256("csv:dataset:" + abs_path)[:32]</span></span></code></span></li>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>excel-connector</span></span></code></span> — <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>sha256(abs_path + "#" + sheet + "#" + table)[:32]</span></span></code></span></li>
<li><span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>postgres-connector</span></span></code></span> — <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>sha256("postgres:dataset:" + host + ":" + port + "/" + db + "/" + schema + "." + table)[:32]</span></span></code></span></li>
</ul>
<h2 id="whats-not-modelled-yet">What's NOT modelled (yet)<a class="anchor" aria-label="Link to section" href="#whats-not-modelled-yet">#</a></h2>
<ul>
<li><strong>Authentication / RBAC</strong> — the API is open in development; production deployments put a proxy in front.</li>
<li><strong>Soft-delete</strong> — deletes are hard; the event log is the recovery path.</li>
<li><strong>Per-relation custom properties</strong> — only <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>:APPLIES_TO</span></span></code></span> carries fields; others are pure edges.</li>
<li><strong>Versioning</strong> — assets don't keep a history of their <span data-rehype-pretty-code-figure=""><code data-language="plaintext" data-theme="github-dark-dimmed github-light"><span data-line=""><span>metadata</span></span></code></span> over time; only the last write wins.</li>
<li><strong>Saved searches / query templates</strong> — every search is stateless.</li>
